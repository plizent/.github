# Support

Thank you for using **Plizent**.

This document explains where to get help, ask questions, report bugs, request features, and report security vulnerabilities for Plizent projects.

---

## Before Requesting Support

Before opening a new issue or discussion:

- Read the project documentation.
- Search existing issues and discussions.
- Make sure your question hasn't already been answered.
- Confirm you're using a supported version of the project, if applicable.

Taking a few minutes to check these resources helps reduce duplicate reports and allows maintainers to focus on new issues.

---

## Support Channels

Use the appropriate channel for your request:

| If you need to... | Use... |
|-------------------|---------|
| Ask a question | GitHub Discussions (if enabled) |
| Report a bug | GitHub Issues |
| Request a feature | GitHub Issues or GitHub Discussions |
| Improve documentation | GitHub Issues or Pull Requests |
| Report a security vulnerability | Follow the instructions in [SECURITY.md](SECURITY.md) |

> [!TIP]
> If a repository provides issue or discussion templates, please use them when creating a new issue or discussion.

---

## Bug Reports

Before reporting a bug, search existing issues to make sure it hasn't already been reported.

When opening a bug report, include:

- A clear description of the problem.
- Steps to reproduce the issue.
- Expected behavior.
- Actual behavior.
- Environment details, such as your operating system, browser, or software version.
- Relevant logs, screenshots, or other supporting information, when available.

Providing complete information helps us understand, reproduce, and resolve issues more quickly.

---

## Feature Requests

We welcome ideas for improving our projects.

Before opening a feature request, search existing issues and discussions to make sure a similar request doesn't already exist.

A good feature request should explain:

- The problem you're trying to solve.
- Your proposed solution.
- Any alternatives you've considered.
- Additional context, examples, or use cases, if helpful.

Clear and well-described feature requests help maintainers understand the problem and make informed decisions.

---

## Community

We are committed to building a welcoming, respectful, and inclusive community.

Please be respectful, constructive, and considerate when interacting with other community members.

All community interactions are governed by our [CODE_OF_CONDUCT.md](CODE_OF_CONDUCT.md).

---

## Response Times

Plizent is maintained by volunteers and contributors.

We do our best to review and respond to questions, issues, and pull requests as time allows. Response times may vary depending on maintainer availability, issue complexity, and project priorities.

> [!NOTE]
> Opening multiple issues or repeatedly requesting updates will not speed up the review process. We appreciate your patience and understanding.

---

## Repository-Specific Support

This document provides the default support guidance for Plizent repositories.

Some repositories may include additional support channels, troubleshooting guides, or project-specific documentation.

When repository-specific support documentation is available, follow that documentation instead of this guide.
